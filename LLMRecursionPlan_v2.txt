The Unified Architecture Blueprint
To combine these four heavyweights, you will build a framework where LangGraph controls the outer loop, ReDel handles the task decomposition, TextGrad optimizes the reasoning, and Aider keeps your 8GB GPU from exploding.

                  +---------------------------------------+
                  |    LANGGRAPH: Deterministic Graph    |
                  +---------------------------------------+
                                      │
                 [State: Init] ➔ [State: Decompose]
                                      │
                                      ▼
                  +---------------------------------------+
                  |      REDEL: Dynamic Tree Spawning     |
                  |  (Spawns Ephemeral Sub-Agent Nodes)  |
                  +---------------------------------------+
                                      │
                         [State: Execute Code / Run Tests]
                                      │
                                      ▼
                  +---------------------------------------+
                  |      AIDER: Context Optimization      |
                  |  (Feeds 1024-token AST Repository Map) |
                  +---------------------------------------+
                                      │
                          [State: Evaluate & Loss Check]
                                      │
                                      ▼
                  +---------------------------------------+
                  |     TEXTGRAD: Textual Backprop       |
                  | (Computes & Feeds Language Gradients) |
                  +---------------------------------------+
                                      │
                       Loops back to fix weights/outputs

================================================================================
  INTERFACE CONTRACTS BETWEEN LAYERS
  Every layer communicates through strictly typed Python dicts (Pydantic
  models at runtime). These contracts are the "nervous system" of the
  project -- get them right first and every phase becomes easier.
================================================================================

Interface 1: LangGraph <--> ReDel
  Direction: LangGraph passes the outer state INTO the decomposition layer;
  ReDel returns spawned child nodes back into the graph state.

  GraphState (dict / Pydantic model):
    task_id          : str        -- unique identifier for the top-level task
    task_description : str        -- natural-language description of the goal
    status           : enum       -- INIT | PLANNING | EXECUTING | VALIDATING | DONE
    children         : [NodeFrame]-- list of spawned sub-agent frames
    depth            : int        -- current recursion depth
    max_depth        : int        -- hard ceiling to prevent over-delegation
    config           : dict       -- model path, VRAM limits, sandbox settings

  ReDel appends new NodeFrame entries to children and returns the mutated
  GraphState back to LangGraph, which routes them to the Execute state.

---

Interface 2: ReDel <--> Aider/Context Layer
  Direction: Spawned sub-agents receive a NodeFrame describing their task;
  they issue ContextRequest messages to the repo-map server to pull only
  the code they need.

  NodeFrame (dict / Pydantic model):
    node_id              : str          -- unique node identifier
    parent_id            : str | None   -- parent node (None for root)
    system_prompt        : str          -- the agent's role instructions
    task                 : str          -- the specific sub-task to perform
    code_context_request : [FilePath]   -- list of file paths the agent wants
    result               : str | None   -- agent's output (filled after execution)
    gradient             : TextGradient | None -- critique (filled after backprop)

  ContextRequest (sent by sub-agent to repo-map server):
    node_id     : str
    file_path   : str
    symbol_name : str | None   -- optional: specific class/function name
    line_range  : (int, int) | None -- optional: specific line span

  ContextPayload (repo-map server reply):
    node_id     : str
    file_path   : str
    content     : str          -- the requested code fragment
    token_count : int          -- how many tokens this fragment costs

---

Interface 3: Aider/Context <--> TextGrad
  Direction: After a sub-agent writes code and it is executed in the sandbox,
  the execution result is handed to TextGrad for evaluation.

  ExecutionResult (dict / Pydantic model):
    node_id     : str
    code_output : str          -- the code the agent wrote
    stdout      : str          -- standard output from execution
    stderr      : str          -- standard error / traceback
    exit_code   : int          -- 0 = success, non-zero = failure
    token_usage : int          -- tokens consumed by this node

  TextGrad receives ExecutionResult, computes a TextGradient, and returns it.

---

Interface 4: TextGrad <--> LangGraph (Feedback Loop)
  Direction: Textual gradients flow backward through the graph to modify the
  prompt or code variable of the node that produced the error.

  TextGradient (dict / Pydantic model):
    node_id           : str          -- which node produced the loss
    loss_description  : str          -- natural-language summary of what failed
    mutations         : [Mutation]   -- ordered list of fixes
      Mutation:
        line       : int           -- line number in the agent's code output
        cause      : str           -- root cause of the error at this line
        suggestion : str           -- concrete code change to apply
    target_variable   : str          -- which variable to mutate ("system_prompt"
                                       or "code_output")
    severity          : float        -- 0.0 (trivial) to 1.0 (fatal)

  LangGraph reads the TextGradient, locates the target node in the graph,
  applies the mutations to its system_prompt or code_output, and re-runs it.

  Note on Phase 3 (TextGrad Backpropagation):
    This interface is intentionally kept mathematically rigorous. The full
    autograd-style textual backprop with dynamic graph tracing is retained
    as a core learning goal. You will implement custom graph traversal code
    that walks the execution tree backward, computes information-density
    scores on critiques (FedTextGrad / UID principle), and routes structured
    gradients to the correct mutable variable. This is the hardest part of
    the project -- and the most rewarding.

---

Interface 5: Harness <--> Recursion Engine
  (See HarnessPlan.txt for the full harness architecture.)

  Direction: The external harness launches the engine, sends commands, and
  subscribes to events for monitoring and visualization.

  HarnessCommand (sent by harness to engine):
    command          : enum  -- RUN | STEP | STATUS | ABORT
    task_description : str | None   -- required when command = RUN
    config           : dict | None  -- optional runtime overrides

  EngineEvent (emitted by engine, consumed by harness):
    event_type : enum -- NODE_SPAWN | NODE_COMPLETE | GRADIENT_FLOW | VRAM_ALERT
    payload    : dict -- event-specific data:
      NODE_SPAWN    : {node_id, parent_id, task}
      NODE_COMPLETE : {node_id, result_summary, token_usage}
      GRADIENT_FLOW : {source_node_id, target_node_id, severity, mutation_count}
      VRAM_ALERT    : {current_vram_mb, threshold_mb, action_taken}

  Communication: Python queue.Queue (threaded) or asyncio.Event (async).
  The harness never touches LLM logic directly -- it only routes data,
  monitors resources, and presents results.

1. The Core Infrastructure (LangGraph + ReDel)
You use LangGraph to design a deterministic state machine (e.g., Start ➔ Plan ➔ Execute ➔ Validate ➔ End).
Inside the Plan state, your Qwen 3.5 9B model acts as the Root Agent. If the task is too large, the model executes a python tool that dynamically appends new child execution nodes to the graph (ReDel style). They don't load new models; they just pass text variables down the chain.

2. The Memory Shield (Aider's Techniques)
Before your spawned agents can even think about writing code, they need context. If you pass an entire codebase to 3 sub-agents, your 8GB VRAM will instant-crash.

The Fix: You implement Aider’s core optimization. Your script will run universal-ctags or an Abstract Syntax Tree (AST) parser over your target directory. It extracts only the class names, function signatures, and variables, and uses a ranking algorithm (like PageRank) to distill your massive codebase into a 1024-token Repository Map. The sub-agents read the map, figure out exactly which file they need, and surgically request only those lines of code.

3. The Self-Correction Engine (TextGrad)
Once a sub-agent writes code, your framework executes it in a local sandbox terminal. If it fails, TextGrad steps in. Instead of just saying "hey, it's broken, fix it," TextGrad creates a tracking graph of your text responses. It treats the terminal error as a mathematical "Loss" function, calculates a "Textual Gradient" (a highly structured, linguistic critique of what went wrong), and backpropagates that gradient through the graph to modify the agent's prompts or code file variable.

Your Milestone Execution Plan (And Where it Gets Hard)
Breaking it down into phases is the only way to survive a massive project like this.

Phase 1: The Spawning Graph (Difficulty: Medium)
Goal: Build the LangGraph state control and write the Python tool that allows the Root Agent to spawn child dictionaries (sub-agents) with custom system prompts.

The Trap: Small models love to over-delegate. Your Qwen model might try to spawn a sub-agent just to add two numbers together. You will have to write strict rules restricting max_depth and enforcing delegation templates.

Phase 2: The VRAM Shield (Difficulty: Medium-High)
Goal: Integrate tree-sitter or ctags to generate the repository maps. Write the logic that allows agents to surgically call an /add [file_name] command to load specific code blocks into their immediate memory buffer, then discard them when finished.

The Trap: Parsing code into token-efficient trees is annoying string-manipulation work. You will spend a lot of time debugging regular expressions and file system paths.

Phase 3: TextGrad Backpropagation (Difficulty: Very High)
Goal: Implement the textual autograd engine. You need to map text strings as mutable variables that can store "text gradients" (LLM-generated evaluations) and pass them backward through your active graph.

The Trap: This is where the difficulty skyrockets. In a static graph, text backprop is straightforward. But because your ReDel layer dynamically spawns child nodes, your TextGrad layer has to dynamically trace the execution tree backward. You will have to write custom graph traversal code in Python to ensure the critique gets routed to the correct system prompt or variable.

These three projects offer highly unique mechanics that you should study and adapt for your project.

1. The RLM (Recursive Language Model) Paradigm
The Project Concept: Inspired by recent research (arXiv:2512.24601), developers have open-sourced RLM architectures (like hampton-io/RLM).

The Unique Feature: It treats long text files or complex tasks as an "external environment" rather than an input prompt. Instead of loading code into the prompt, the model writes lightweight execution scripts (like a tiny local JavaScript or Python REPL) to recursively query, split, and search chunks of data via sub-calls to itself.

How to steal it for LLMRecursive: When your ReDel layer spawns a sub-agent to review code, don't let it output chat prose. Force the sub-agent to strictly communicate with its parent node via a local JSON execution frame. The sub-agent runs its checks, dumps its structural metrics into a tiny JSON string, and terminates. This entirely bypasses the chat "text bottleneck" and saves thousands of tokens.

2. OpenViking (Tiered Context Loading)
The Project: volcengine/OpenViking is an open-source context database built for local agent harnesses (like OpenClaw).

The Unique Feature: It replaces standard vector database memory with a File System Paradigm using a Three-Tier Context Structure (L0 / L1 / L2).

L0 (Immediate Brain): Raw variables and the exact 5 lines of code being edited right now.

L1 (Local Cache): A tree-sitter representation of the current file's functions.

L2 (Disk Storage): The rest of the codebase tucked away in a local SQLite file.

It dynamically promotes and demotes data between these tiers automatically based on what the agent is looking at, compressing old conversation turns into summarized "long-term memory blocks" on the fly.

How to steal it for LLMRecursive: This is your silver bullet for the 8GB VRAM limit. You can code a mini python manager inside your project that acts as the "VRAM gatekeeper." If your local model’s active KV cache exceeds 1.5 GB, the gatekeeper automatically sweeps the oldest sub-agent history out of VRAM (L0), serializes it to a markdown file on your hard drive (L2), and leaves behind a simple 1-sentence text summary (L1).

3. Repomix (Tree-Sitter XML Packing)
The Project: Repomix (which exploded to over 22k stars on GitHub) is an alternative approach to Aider's repository mapping.

The Unique Feature: Instead of relying strictly on PageRank algorithms to choose files, Repomix flattens entire directories into raw, highly compressed XML-structured files using tree-sitter. It achieves an astronomical 70% token reduction while keeping the exact scoping, classes, and loops intact.

How to steal it for LLMRecursive: Aider's graph-based PageRank can be mathematically heavy to implement from scratch. Instead, build your codebase parser using the Repomix philosophy. Wrap your Qwen 3.5's input data in custom XML blocks like <codebase_summary> and <file path="main.py">. Smaller models like Qwen are incredibly good at parsing XML tags because the explicit open/close structures anchor their attention mechanisms, reducing hallucinations.

4. FedTextGrad (Linguistic Gradients Aggregation)
The Project: ubc-tea/FedTextGrad (from ICLR). It explores how to aggregate text gradients across multiple sources.

The Unique Feature: It utilizes the Uniform Information Density (UID) principle. When multiple agents or steps compute text feedback (gradients), the text can become messy and contradictory. FedTextGrad uses an information-theoretic approach to force the LLM to compress critiques down to their highest-density data points, stripping out conversational fluff.

How to steal it for LLMRecursive: When your TextGrad layer computes a textual "loss" (a critique of a broken function), don't pass a raw paragraph back to the code model. Write a strict regular expression validator that forces the TextGrad evaluator to format its gradient in a tightly packed structure: [Line Number] -> [Error Cause] -> [Suggested Mutation].

The Game Plan for Your Project Shape
By combining these open-source secrets, the underlying engine of your summer project will look like a highly refined data pipe:

Repomix XML-packing prepares your code structure at a 70% token discount.

OpenViking's Tiered System ensures your VRAM never crosses your 8GB threshold by swapping old memories to your hard drive.

RLM execution blocks handle the recursive sub-agent loops silently in the background.

FedTextGrad principles ensure your error feedback loops are mathematically dense, crisp sentences rather than wordy chat loops.

This shifts the project away from standard, generic "AI prompting" and turns it into deep, low-level context engineering.
